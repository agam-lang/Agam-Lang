# Phase P1 — WASM Backend

**Status:** open
**Tier:** 3 (Platform and Ecosystem Breadth)

## Scope

Add WebAssembly as a compilation target, enabling browser-based execution (web playground), edge computing, and WASI-based server-side WASM.

## Deliverables

### WASM Codegen
- [ ] `wasm32-wasi` target in `agam_codegen` (via LLVM wasm backend)
- [ ] `wasm32-unknown-unknown` for browser embedding
- [ ] `agamc build --target wasm32-wasi` integration
- [ ] WASM-compatible subset of runtime (no OS-level syscalls in browser)
- [ ] Linear memory management for WASM target

### WASI Support
- [ ] WASI preview 1 compliance for stdio, filesystem, clock, random
- [ ] Server-side WASM execution (Wasmtime, Wasmer compatibility)
- [ ] `agamc run --target wasm32-wasi` via bundled WASM runtime

### Browser Integration
- [ ] JavaScript interop layer for calling JS from Agam and vice versa
- [ ] DOM access utilities (stretch — may be library-level)
- [ ] WASM binary size optimization (`wasm-opt` integration)

### Size Optimization
- [ ] Dead code elimination for WASM output
- [ ] `--release-size` profile for minimal WASM binaries
- [ ] Tree-shaking at the function level

## Responsible Crates

- `agam_codegen` — WASM target codegen
- `agam_runtime` — WASM-compatible runtime subset
- `agam_driver` — `--target wasm32-*` flag handling

## Dependencies

- Phase F2/F3 (type system, object model) — WASM output needs stable type layouts
- LLVM WASM backend (already mature in LLVM)
